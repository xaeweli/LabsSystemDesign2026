##Архитектура системы
Архитектура системы описана с использованием Structurizr DSL и включает несколько уровней: System Context, Container и Dynamic диаграммы.

###System context

Основным актором системы является пользователь. Пользователь взаимодействует с системой менеджера рецептов через клиентское приложение.
Система представляет собой API для работы с рецептами, профилем пользователя.

Система вазимодействует с внешними сервисами, для произведения аутентификации пользователя:
- Email сервис для отправки писем для регистрации
- Facebook сервис для аутентификации через Facebook
- Google сервис для аутентификации через Google

###Container Диаграмма

Система разделена на несколько контейнеров:
- WebApp - клиентское веб приложение, через которые пользователь работает с системой
- API Gateway - единая точка входа для всех запросов клиента
- UserAuth Service - сервис аутентификации пользователей, вход в существующий аккаунт, создание нового
- Userprofile Service - управление профилем пользователя, чтение профилей других пользователей
- Recipepage Service - сервис отвечающий за предоставление информации о конкретном рецепте, добавление рецепта в избранное
- Create and Edit Recipe Service - создание нового рецепта, изменение старого
- Search Service - поиск пользователей и рецептов, по названию ингридиентам и т.д.
- Redis Cache - кэширование часто запрашиваемых данных, хранение временных данных (незаконченный рецепт)
- Recipes Database - хранение информации о рецептах (id, название, описание, ингридиенты, автор, кол-во добавленных в избранное)
- Users Database - хранение информации о пользователях (id, юзернейм, имя, фамилия, аватар, сохраненные рецепты, добавленные рецепты)
- User Auth Database - хранение информации о пользователях необходимой для аутентификации (id, логин, пароль, email и т.д.)
  
Сервисы взаимодействуют между собой через REST API

##Основные задачи пользователей

В системе существуют контейнеры для решения следующих задач пользователей.

###Регистрация
1. Пользователь отправляет запрос на регистрацию через клиентское приложение.
2. Запрос поступает в API Gateway.
3. API Gateway передает запрос в UserAuth Service.
4. UserAuth Service перенаправляет на внешние сеервисы, для продолжения регистрации.
5. После подтверждения от внешних сервисов данные добавляются в Users Database и User Auth Database.

###Добавление нового рецепта 
1. Пользователь отправляет запрос на добавление новго рецепта.
2. Запрос поступает в API Gateway.
3. API Gateway передает запрос в Create and Edit Recipe Service.
4. Create and Edit Recipe Service загружает временные данные в cache.
5. Create and Edit Recipe записывает данные о новом рецепте в Recipes Database и добавляет id рецепта в список добавленных пользователем в Users Database.

###Другие задачи пользователей реализованные системой
- Поиск пользователей по логину, имени, фамилии.
- Поиск рецептов по названию, ингридиентам и описанию.
- Изменение существующего рецепта, добавление новых ингридиентов
- Получение информации о рецепте.
- Получение информации о пользователе, список рецептов добавленных пользователем.
- Добавление рецепта в избранное.
