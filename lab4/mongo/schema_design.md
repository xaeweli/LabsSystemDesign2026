# Проектирование документной модели MongoDB

## Вариант 23 — Система управления рецептами

## Обоснование выбора между Embedded и References

### Связь Recipe → Ingredient (Embedded)

Ингредиенты **встраиваются** в документ рецепта.

**Обоснование:**
- Ингредиенты всегда запрашиваются вместе с рецептом
- Количество ингредиентов на один рецепт невелико (обычно 5–20)
- Нет необходимости запрашивать ингредиенты независимо от рецепта
- Ингредиенты не используются отдельно от рецепта в других контекстах
- Одно обращение к БД вместо JOIN/ lookup

### Связь User → Recipe (References)

Рецепты пользователя хранятся как **references** (массив `recipe_ids`).

**Обоснование:**
- Рецепты — самостоятельные сущности, могут запрашиваться отдельно
- Один рецепт связан с одним пользователем (author), но пользователь может иметь много рецептов
- Размер массива recipe_ids может быть большим
- Рецепты обновляются независимо от профиля пользователя

### Связь User → Favorites (References)

Избранные рецепты хранятся как **references** (массив `favorite_recipe_ids`).

**Обоснование:**
- Многие пользователи могут добавлять один и тот же рецепт в избранное
- Рецепты — самостоятельные сущности
- Массив избранного может быть большим
- Не требуется каскадное обновление/удаление при изменении рецепта

### Связь User → Credentials (Embedded)

Данные аутентификации объединены с профилем пользователя.

**Обоснование:**
- При создании пользователя создаются и профиль, и учётные данные
- Запрос профиля часто требует только базовой информации
- При поиске по логину нужны и credentials, и профиль

## Документная модель

### Коллекция `recipes`

```json
{
  "_id": ObjectId,
  "title": "Борщ",
  "description": "Традиционный русский суп",
  "instructions": "1. Сварить бульон... 2. Добавить овощи...",
  "cooking_time_minutes": 90,
  "author_id": ObjectId,
  "favorite_count": 42,
  "created_at": ISODate,
  "ingredients": [
    {
      "name": "Свёкла",
      "amount": "2",
      "unit": "шт"
    },
    {
      "name": "Картофель",
      "amount": "3",
      "unit": "шт"
    }
  ]
}
```

**Индексы:**
- `{ title: 1 }` — поиск и сортировка по названию
- `{ author_id: 1 }` — получение рецептов пользователя
- `{ title: "text" }` — полнотекстовый поиск

---

### Коллекция `users`

```json
{
  "_id": ObjectId,
  "username": "chef_ivan",
  "password_hash": "hash_123456",
  "email": "ivan@example.com",
  "first_name": "Иван",
  "last_name": "Петров",
  "recipe_ids": [ObjectId, ObjectId],
  "favorite_recipe_ids": [ObjectId, ObjectId]
}
```

**Индексы:**
- `{ username: 1 }` — уникальный, поиск по логину
- `{ email: 1 }` — уникальный
- `{ first_name: 1, last_name: 1 }` — поиск по маске имени/фамилии

## Сравнение с реляционной моделью (лабораторная 3)

| Реляционная модель (lab3)        | Документная модель MongoDB (lab4) |
|----------------------------------|-----------------------------------|
| 3 отдельных БД, 6 таблиц         | 2 коллекции                      |
| recipes + ingredients (2 таблицы)| recipes (с встроенными ingredients)|
| user_profiles + user_credentials | users (объединённый профиль)     |
| user_recipes (таблица связей)    | users.recipe_ids (массив)         |
| user_favorites (таблица связей)  | users.favorite_recipe_ids (массив)|
| JOIN для получения ингредиентов  | embedded — одно чтение           |
| JOIN для связей user-recipe      | lookup / два запроса             |

### Преимущества документной модели для данного проекта

1. **Меньше запросов** — ингредиенты читаются за одно обращение
2. **Меньше сущностей** — 2 коллекции вместо 3 БД / 6 таблиц
3. **Естественная вложенность** — ингредиенты не существуют без рецепта
4. **Гибкость схемы** — легко добавлять поля без миграций
5. **Атомарность** — обновление рецепта с ингредиентами — одна операция
