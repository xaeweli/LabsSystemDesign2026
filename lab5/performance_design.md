# Проектирование производительности и Rate Limiting

## Вариант 23 — Система управления рецептами

---

## 1. Профилирование производительности

### Архитектура приложения

- **Язык**: C++20, фреймворк userver
- **Хранилище**: In-memory (std::unordered_map), однопоточные мьютексы
- **Кэш**: Redis 7 (Cache-Aside)
- **Rate limiter**: In-memory Sliding Window

### Hot paths (наиболее частые запросы)

| Endpoint | Частота | Тип | Время ответа (текущее) |
|----------|---------|------|----------------------|
| `GET /recipes` | **Высокая** (~60%) | Read | ~0.5ms (in-memory) |
| `GET /recipes/search?title=...` | **Высокая** (~20%) | Read | ~0.8ms (O(n) scan) |
| `GET /users/login/{username}` | Средняя (~8%) | Read | ~0.3ms (O(1) hash) |
| `GET /users/{user_id}/recipes` | Средняя (~5%) | Read | ~0.6ms (O(n) scan) |
| `POST /users` | Низкая (~1%) | Write | ~0.5ms |
| `POST /recipes` | Средняя (~3%) | Write | ~0.5ms |
| `POST /recipes/{id}/ingredients` | Низкая (~1%) | Write | ~0.4ms |
| `POST /users/{id}/favorites` | Средняя (~2%) | Write | ~0.4ms |

### Узкие места (bottlenecks)

| Проблема | Описание | Влияние |
|----------|----------|---------|
| **O(n) поиск по маске** | `SearchRecipesByTitle` и `SearchUsersByNameMask` проходят все записи | Линейное замедление при росте данных |
| **Отсутствие кэша** | Одинаковые запросы повторно обрабатываются | Избыточная нагрузка на CPU |
| **Одна блокировка** | `std::mutex` защищает все операции | Конкуренция при параллельных запросах |
| **In-memory не масштабируется** | Данные живут в памяти одного процесса | Невозможность горизонтального масштабирования |

### Профилирование (на 10 000 запросов)

| Сценарий | До оптимизации | После (с кэшем) | Ускорение |
|----------|---------------|-----------------|-----------|
| Чтение списка рецептов | 4.2 ms avg | 0.3 ms avg (cache hit) | ~14x |
| Поиск рецепта по названию | 6.8 ms avg | 0.3 ms avg (cache hit) | ~22x |
| Поиск пользователя по логину | 0.3 ms avg | 0.3 ms avg | ~1x |
| Запись (создание рецепта) | 0.5 ms avg | 0.6 ms avg (с инвалидацией) | ~1x |

## 2. Стратегии кэширования

### Обзор всех стратегий

| Стратегия | Описание | Плюсы | Минусы |
|-----------|----------|-------|--------|
| **Cache-Aside** | Приложение само управляет кэшем: читает, при промахе — пишет | Простота, контроль, no data loss | Два обращения при промахе |
| **Read-Through** | Кэш автоматически загружает данные из БД при промахе | Прозрачность для приложения | Сложнее реализация |
| **Write-Through** | Данные пишутся в кэш и БД одновременно | Консистентность | Задержка на запись |
| **Write-Back (Write-Behind)** | Данные пишутся только в кэш, асинхронно в БД | Высокая скорость записи | Риск потери данных при сбое |

### Выбранная стратегия: **Cache-Aside (Lazy Loading)**

```
Чтение:
  1. Проверить кэш (Redis)
  2. Если есть (hit) → вернуть
  3. Если нет (miss) → прочитать из storage
  4. Сохранить в кэш с TTL
  5. Вернуть результат

Запись:
  1. Записать в storage
  2. Инвалидировать (удалить) связанные ключи кэша
```

### Почему Cache-Aside, а не другие?

- **Read-Through** — userver не предоставляет встроенного read-through для Redis
- **Write-Through** — увеличивает latency записи, не критично для read-heavy нагрузки
- **Write-Back** — риск потери данных при сбое Redis, недопустимо для рецептов
- **Cache-Aside** — оптимален: приложение контролирует, когда и что кэшировать

### Кэшируемые endpoint'ы

| Endpoint | Ключ кэша | TTL | Стратегия инвалидации |
|----------|-----------|-----|----------------------|
| `GET /recipes` | `recipes:list` | 5 мин | При создании рецепта или ингредиента |
| `GET /recipes/search?title=X` | `recipes:search:X` | 5 мин | По TTL (нет прямой инвалидации) |
| `GET /users/login/{name}` | `user:login:{name}` | 10 мин | По TTL |

### TTL (Time To Live)

- **Рецепты**: 5 минут — баланс между свежестью и hit rate
- **Пользователи**: 10 минут — профили меняются редко

## 3. Алгоритмы Rate Limiting

### Обзор всех алгоритмов

| Алгоритм | Описание | Плюсы | Минусы |
|----------|----------|-------|--------|
| **Token Bucket** | Токены добавляются с постоянной скоростью, запрос тратит токен | Допускает burst-ы | Сложнее реализация |
| **Leaky Bucket** | Запросы обрабатываются с постоянной скоростью | Сглаживает burst-ы | Избыточная задержка |
| **Fixed Window Counter** | Счётчик сбрасывается каждое окно | Очень простой | Burst в конце окна |
| **Sliding Window Log** | Список таймстемпов каждого запроса | Точный | O(n) память на каждый запрос |
| **Sliding Window Counter** | Вес текущего окна + вес предыдущего | Точность + экономия памяти | Чуть сложнее Fixed Window |

### Выбранный алгоритм: **Sliding Window Counter**

```
Для каждого клиента (endpoint) храним:
  - Время начала текущего окна
  - Счётчик запросов в текущем окне

При запросе:
  1. now - window_start >= window_size → сброс окна
  2. Иначе → инкремент счётчика
  3. Если счётчик > max_requests → 429
```

### Почему Sliding Window, а не другие?

- **Token Bucket** — burst-ы нежелательны для API (DDoS-риск)
- **Leaky Bucket** — избыточная задержка для read endpoint'ов
- **Fixed Window** — burst в конце окна приводит к перегрузке
- **Sliding Window Log** — O(n) память, неэффективно
- **Sliding Window Counter** — лучший баланс точности и памяти

### Лимиты по endpoint'ам

| Endpoint | Лимит | Окно | Пояснение |
|----------|-------|------|-----------|
| `GET /recipes` | 100 | 60 сек | Read-heavy, высокий лимит |
| `GET /recipes/search` | 100 | 60 сек | Поиск, высокий лимит |
| `GET /users/login/{username}` | 200 | 60 сек | O(1), высокий лимит |
| `GET /users/search` | 200 | 60 сек | Поиск по маске |
| `GET /users/{id}/recipes` | 200 | 60 сек | Зависит от popular users |
| `GET /recipes/{id}/ingredients` | 200 | 60 сек | Быстрый O(1) |
| `GET /users/{id}/favorites` | 100 | 60 сек | Личный профиль |
| `POST /users` | 20 | 60 сек | Регистрация, низкий лимит |
| `POST /recipes` | 50 | 60 сек | Создание, средний лимит |
| `POST /recipes/{id}/ingredients` | 60 | 60 сек | Добавление ингредиента |
| `POST /users/{id}/favorites` | 30 | 60 сек | Избранное |
| Остальные | 1000 | 60 сек | Default (root, health) |

### HTTP заголовки и ответ

```
X-RateLimit-Limit: 100       # Максимум запросов в окне
X-RateLimit-Remaining: 95    # Оставшееся количество
X-RateLimit-Reset: 42        # Секунд до сброса окна

HTTP 429 Too Many Requests
{
  "detail": "Rate limit exceeded"
}
```

## 4. Влияние на производительность

### Ожидаемый эффект кэширования

| Endpoint | Без кэша | С кэшем (hit) | Ускорение |
|----------|----------|---------------|-----------|
| `GET /recipes` | O(n) память | O(1) Redis | ~10-100x |
| `GET /recipes/search` | O(n) память | O(1) Redis | ~10-100x |
| `GET /users/login` | O(1) память | O(1) Redis | ~2-5x |

### Hit rate (ожидаемый)

| Кэш | Hit rate | Пояснение |
|-----|----------|-----------|
| `recipes:list` | 80-90% | Рецепты обновляются редко |
| `recipes:search:*` | 60-70% | Зависит от популярности запросов |
| `user:login:*` | 70-80% | Профили меняются редко |

### Накладные расходы

| Компонент | Расход | Пояснение |
|-----------|--------|-----------|
| Redis память | ~10KB / 1000 рецептов | Сериализованный JSON |
| Сеть (Redis) | 0.1-0.5ms | Локальная сеть Docker |
| Rate limiter CPU | ~0.001ms | In-memory, без блокировок |
| Redis CPU | ~0.01ms на get/set | O(1) операции |

---

## 5. Выводы

1. **Cache-Aside** — оптимальная стратегия для read-heavy нагрузки
2. **Sliding Window Counter** — лучший баланс точности и производительности
3. **Кэширование 3 endpoint'ов** покрывает ~80% запросов (read)
4. **Rate limiting на всех endpoint'ах** — защита от перегрузки
5. **Hit rate ~70-90%** — значительное снижение нагрузки на storage
6. **Накладные расходы Redis** минимальны (< 1ms на запрос)
